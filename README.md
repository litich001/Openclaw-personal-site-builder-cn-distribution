# Personal Site Builder CN

一个中文优先的个人网站生成 skill。  
说得更直白一点：它是用来把“我有一份简历，但我不想再把它丢给别人看一份毫无记忆点的 PDF”这件事，变成一个真正像样的网站。

如果你也受不了那种：

- 内容很多，但看不出重点
- 页面很满，但没有气质
- 动效很多，但完全不适合求职
- 手机一打开就崩
- SEO 一塌糊涂
- 上线步骤复杂得像在报税

那么这套 skill，就是专门冲着这些问题来的。

---

## 它到底是干什么的

这不是一个“给你一套模板，你自己填空”的普通网页工具。  
它更像是一套给 AI coding agent 的工作方法，专门用来做这件事：

**把简历、项目、照片、证书、联系方式等资料，整理成一个中文优先、适合求职展示、设计在线、移动端正常、SEO 基础完整、还能真的发到公网的网站。**

换句话说，它不只关心“页面有没有做出来”，还关心：

- 面试官打开之后能不能快速看懂你是谁
- 页面是不是有质感，而不是廉价模板味
- 手机上会不会直接翻车
- 联系方式和简历入口是不是足够清楚
- 图片和站点资源是不是能正常加载
- 上线之后是不是一个能发给别人的链接，而不是停在本地自娱自乐

---

## 这个 skill 适合谁

如果你属于下面这些人，这套东西大概率会很有用：

### 1. 想给自己做个人网站的人
你有简历，也有项目，甚至还有一些照片、证书、活动海报。你不缺内容，缺的是一个真正能把这些内容组织好、做得体面的方式。

### 2. 想帮别人做网站的人
你是自由职业者、朋友里的“技术担当”、或者团队里总是被委托“顺手做个网站”的那个人。你不想每次都从零开始想结构、想文案、想部署。

### 3. 想把这件事交给 AI 代理自动跑的人
你希望本地代理能自己读资料、自己梳理、自己生成站点、自己打包，最好还能顺手部署出去。你不想每次都从头解释一遍“这个按钮别太大、二维码别做成墓碑、移动端别崩”。

### 4. 手里资料并不完整的人
你可能还没有一份特别工整的最终简历，甚至只有 PDF、几张图、几段项目介绍。没关系，这套 skill 也支持“先问答补齐，再生成第一版”。

---

## 它能帮你做到什么程度

如果目标代理能力足够，这套 skill 理论上可以跑完整条链路：

1. 读取用户本地资料目录
2. 判断资料是否齐全
3. 缺什么就问什么
4. 把零散信息整理成网站结构
5. 生成站点首页、资料展示、联系页、网页版简历页
6. 同时处理设计、SEO、移动端和交互
7. 输出一个可直接上传的静态站目录
8. 在具备权限时部署到 Cloudflare 等平台
9. 回填真实线上地址到 SEO 字段
10. 交付一个真的能访问的网址

这不是“理论上的可能”，而是这套 skill 设计时就明确围绕这条链路来收口的。

---

## 这套 skill 特别在意什么

很多“个人网站生成器”最后会做成两种东西：

- 一种像 PPT 上墙，信息堆得满满当当，但没有重点
- 一种像模板市场样板房，看起来没毛病，但谁都不像

这套 skill 刻意绕开了这两个坑。

它优先强调这些：

### 中文语境优先
不是把英文 portfolio 站生硬翻译成中文，而是按中文阅读和中文招聘场景来组织信息。

### 求职与职业展示优先
默认目标读者是 HR、面试官、合作方，而不是设计奖评委。好看要有，但阅读效率更重要。

### 桌面端和移动端都要成立
桌面端不能散，移动端不能炸。尤其移动端，不允许标题失控、按钮拥挤、导航压坏布局。

### 不允许大面积无意义空白
页面的留白应该服务节奏，而不是让人怀疑是不是还有内容没加载出来。

### 不允许 AI 味过重的文案
网站不是来朗诵“赋能、协同、引领、闭环、重构”的。文案要像人说的话，而且要像这个用户会说的话。

### 不默认迷信重型动效
动画可以有，但必须服务表达。酷炫不是让页面打开像要起飞，而是让用户觉得“这人做事是有审美、有控制力的”。

## 它支持什么输入方式

### 方式一：完整资料直投
适合已经准备好了这些内容的人：

- 简历 PDF / Word / 文本
- 项目经历
- 技能栈
- 联系方式
- 照片
- 证书
- 海报 / 活动图
- 外部链接

这种情况下，代理可以直接开始整理结构和建站。

### 方式二：资料不全，先问答补齐
如果用户资料不完整，skill 不应该直接僵住，而是会优先补齐这些问题：

- 想找什么岗位
- 有多少年经验
- 最想突出哪些优势
- 哪几段经历最重要
- 哪些联系方式愿意公开
- 希望什么样的风格

这意味着：**没有完整简历，也能先做第一版。**

## 这个仓库里有什么

```text
personal-site-builder-cn-distribution/
├─ README.md
├─ USAGE.md
├─ OPENCLAW-NOTES.md
├─ AGENT-RUNBOOK.md
└─ skill/
   └─ personal-site-builder-cn/
      ├─ SKILL.md
      ├─ agents/
      │  └─ openai.yaml
      └─ references/
         ├─ workflow.md
         └─ langgpt-prompt.md
```

### 文件作用速览

- `README.md`
  你现在正在看的这份说明。适合放 GitHub 首页，给人快速建立预期。
- `USAGE.md`
  写给普通使用者。告诉别人怎么安装、怎么调用、怎么给代理下任务。
- `OPENCLAW-NOTES.md`
  写给打算接 OpenClaw 或其他自动化代理的人。重点讲清楚能力边界和适配前提。
- `AGENT-RUNBOOK.md`
  写给自动化代理的执行说明。比 prompt 更像“执行协议”，告诉代理按什么顺序推进、失败时怎么回退。
- `skill/personal-site-builder-cn/SKILL.md`
  核心规则文件。定义这套 skill 的工作边界、设计标准和输出要求。
- `workflow.md`
  工作流定义。适合代理按照阶段逐步推进，不会一上来就莽进去写页面。
- `langgpt-prompt.md`
  对话起始 prompt。适合直接复制给代理，用于快速进入角色。

## 怎么分享给别人

最简单就是两种方式：

### 方式一：上传到 GitHub
适合长期公开维护，也方便别人 fork 或下载。

### 方式二：打包成 zip 发给别人
适合你先在小范围内测试，看看不同代理接进去之后效果怎么样。

别人拿到后，一般有两种使用方式：

1. 把 `skill/personal-site-builder-cn` 放进自己的本地 skill 目录
2. 如果工具不支持本地 skill，就手动读取里面的 `SKILL.md + workflow.md + langgpt-prompt.md`

---

## 这里有一个非常重要的现实说明

**这不是“上传到 GitHub 就自动生效”的通用插件。**

它能不能被真正调用，取决于目标工具有没有下面这些能力：

- 本地 skill 目录
- 外部 prompt / workflow 注入
- 从仓库安装或导入 skill
- 本地文件读取
- 文件编辑
- 命令执行
- 静态目录打包
- 上传部署

所以更准确地说：

- 如果对方工具能力足够，这套 skill 可以端到端工作
- 如果对方工具能力一般，它仍然可以作为一套非常完整的规则和 prompt 来参考

## 它特别适合接谁

### Codex / 本地 coding agent
最自然，尤其适合能读本地文件、改文件、跑命令、做部署的环境。

### Claude Code / 同类代理
前提一样：如果支持本地上下文和文件操作，就能很好复用。

### OpenClaw
可以接，但要更严谨一点。因为 OpenClaw 是否能完整跑通，取决于它有没有：

- 本地文件读取能力
- 命令执行能力
- 编辑文件能力
- 打包目录能力
- 浏览器/上传能力

如果缺其中几项，它可以“参考这套 skill”，但不一定能“完整自动执行”。

详细见 [OPENCLAW-NOTES.md](./OPENCLAW-NOTES.md)。

## 推荐的资料目录长什么样

如果你想让代理更顺手，建议用户资料按这种方式整理：

```text
project-root/
├─ materials/
│  ├─ resume.pdf
│  ├─ intro.txt
│  ├─ links.txt
│  ├─ photos/
│  ├─ posters/
│  └─ certs/
├─ output-site/
│  ├─ site/
│  ├─ deploy/
│  └─ archives/
└─ notes/
```

为什么推荐这样做？

因为一旦目录清楚，代理就不需要先花很多轮时间去猜：

- 图片在哪
- 证书在哪
- 项目经历是哪个文件
- 最终输出应该放哪

而一旦代理开始猜，结果通常就会越来越有“玄学调参”的味道。我们尽量别给它这个发挥空间。

---

## 默认产出长什么样

这套 skill 默认更倾向于输出一个**静态站目录**，而不是一堆零散 HTML 片段。

推荐的结果结构一般像这样：

```text
site/
├─ index.html
├─ resume.html
├─ styles.css
├─ script.js
├─ robots.txt
├─ sitemap.xml
├─ site.webmanifest
└─ assets/
```

这么做的好处很简单：

- 可直接上传
- 易于分发
- 易于给别人继续维护
- 也更适合自动化代理接手下一步部署

---

## 它写网站时默认会守住哪些边界

这是这套 skill 很核心的一部分。因为很多失败的网站，不是“不会写”，而是“没人拦着它乱写”。

默认会守的边界包括：

- 不允许大面积无意义空白
- 不允许标题断词
- 不允许 AI 味文案泛滥
- 不允许联系页做成纪念碑广场
- 不允许二维码大卡片比内容还抢戏
- 不允许样式规则越改越乱，最后用几十层覆盖互相打架
- 不允许移动端只做“缩小版桌面端”

简单说，这套 skill 想做的是：

**不是随便做一个站，而是做一个你愿意真正发给别人的站。**

## 一句话总结

如果你想找的不是一个“看起来像网站”的模板，而是一套能把简历资料稳稳落成网站、还能继续扩展到部署和自动化的工作方法，那么这套 skill 就是给这个目标准备的。
