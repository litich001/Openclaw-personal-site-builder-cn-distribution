# 工作流

## 第一阶段：明确目标与约束

1. 确认受众：面试官、HR、招聘方、客户，还是泛公众。
2. 确认主目标：求职、作品展示、个人品牌建设，或混合目标。
3. 确认交付约束：预算为零或极低、操作简单、中国大陆可访问、是否需要双语、希望多快上线。
4. 先明确核心取舍：免费和简单通常可以做到，但如果要中国大陆长期稳定且更快，后续往往需要升级基础设施。
5. 明确文案边界：哪些内容必须保留用户原话，哪些内容只允许压缩和排版，不允许擅自改写。
6. 如果用户资料不全，先转入问答式采集，不要卡住整个流程。

## 第二阶段：把原始材料转换成网页内容

1. 接收用户的简历、简介、项目、链接、照片、PDF、本地目录等资料。
2. 如果用户没有完整材料，则通过问答补齐：岗位方向、经历、项目、技能、联系方式、风格偏好。
3. 整理成结构化内容模型：
   - 首屏 hero
   - 个人定位 about
   - 经历 experience
   - 项目 projects
   - 资料展示 gallery
   - 联系方式 contact
   - 网页版简历 resume（可选但推荐）
4. 把偏长的简历表述压缩成适合网页阅读的文案层级。
5. 标出敏感信息，并给出更安全的公开替代方案。
6. 明确哪些字段需要进入 TDK 与结构化数据。

## 第三阶段：调研参考并选择视觉方向

1. 当用户要求参考站或最新灵感时，搜索当前高质量个人网站案例。
2. 按这些标准筛选参考：
   - 首屏是否清晰
   - 阅读是否顺畅
   - 手机端是否成立
   - 是否适合中国大陆访问环境
   - 实现成本是否合理
3. 给出 2 到 3 个视觉方向，并说明各自取舍。
4. 默认优先推荐“高质感、科技感、简历可读、非模板化”的方向。
5. 先确定一个方向，再进入实现。

## 第四阶段：确定信息架构与交互范围

1. 固定站点结构，是单页还是多页。
2. 桌面端首屏默认采用左右布局，移动端再自然折叠，避免内容漂浮或上下堆叠失衡。
3. 决定是否需要：
   - 单页结构
   - 多页结构
   - 双语结构
   - 网页版简历页
4. 只定义必要交互，例如：
   - 开场转场
   - 分段入场
   - 平滑滚动过渡
   - 悬停反馈
   - 分类切换与图片放大
   - 电话/简历悬浮快捷操作
5. 拒绝影响招聘阅读效率或移动端性能的重型效果。
6. 明确禁止项：大面积无意义留白、过高联系页、过大二维码卡片、说明性过程文案、断裂标题、互相冲突的重复样式覆盖。

## 第五阶段：以静态优先方式实现

1. 主体内容优先使用静态渲染。
2. 尽量使用本地或自托管资源。
3. 避免依赖不稳定的外部字体、视频、分析脚本或嵌入内容。
4. 保持内容结构可维护，确保后续改资料时不需要重做设计。
5. 推荐将简历输出为网页版页面，而不是只提供 PDF 下载。
6. 实现时按页面逐屏检查桌面端与移动端：
   - 首页是否撑满且左右平衡
   - 个人定位是否阅读顺畅
   - 职业经历是否可收展
   - 资料展示是否分类明确
   - 联系页是否紧凑且无空洞
   - 移动端首屏是否可读、可点、可滚动
   - 网页版简历是否适合手机直接打开
7. 每一轮样式调整后要收口，不继续叠加失控媒体查询。

## 第六阶段：补齐 SEO 和性能优化

1. 添加页面级 metadata。
2. 添加 OG 标签和分享图策略。
3. 添加 sitemap、robots、canonical 和结构化数据。
4. 优化图片、字体和动画路径。
5. 测试响应式布局和移动端加载表现。
6. 同步用户最终确认的网站名称、Title、Description 和 Keywords。
7. 如果最终线上地址发生变化，及时同步修改 canonical、og:url、sitemap 等字段。

## 第七阶段：务实上线

1. 先使用免费公网静态托管上线。
2. 上线后验证：
   - 构建是否成功
   - 公网链接是否可访问
   - 首页开场动画是否可进入
   - 简历页是否正常打开
   - 图片是否稳定加载
   - 手机端是否可读且流畅
3. 如果使用 Cloudflare，优先准备“无需构建、可直接上传”的静态目录。
4. 如果用户后续要求更强的大陆访问稳定性，再解释 ICP 备案与迁移路径。

## 第八阶段：给别人或自动化代理复用

1. 要求使用者把资料放进单独项目目录，避免素材散落。
2. 如果资料不完整，先执行问答补齐，再落地第一版。
3. 默认输出静态目录，保证他人或代理无需额外构建即可上传。
4. 如果要给自动化代理，例如小龙虾，优先走：读取本地文件 -> 梳理内容 -> 生成站点 -> 打包静态目录 -> 上传 Cloudflare -> 回填真实线上地址。
5. 如果目标代理无法执行上传，则退化为：生成代码 + 生成部署说明 + 等待人工上传。

## 第九阶段：最终交付

交付内容包括：

- 公网访问地址
- 可编辑内容所在位置
- 部署说明
- TDK 最终内容
- 可分享给第三方复用的技能与提示词
- 下一步升级建议

推荐升级方向：

- 自定义域名
- 中英双语扩展
- 更适合大陆访问的统计方案
- 完成备案后的国内托管迁移
- 给自动化代理接入本地文件读取、打包和 Cloudflare 上传流程
